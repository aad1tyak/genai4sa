Empirical Acquisition and Extraction of Software Architecture Datasets: Exact Repositories, Scripts, and Evaluation FrameworksThe demand for grounded, actionable data in the realm of empirical software engineering necessitates a strict departure from abstract theoretical frameworks. When constructing large language model (LLM) benchmarks, training corpora, or analytical tools for software architecture, data engineers and researchers require exact repository locations, precise file paths, explicit dataset links, and the functional scripts necessary to acquire and parse this information. The transition to generative-oriented software development introduces severe risks of architectural degradation, driven not merely by the quality of individual code generation outputs, but by the absence of mechanisms for maintaining architectural invariants and causal relationships between requirements, decisions, and their implementation.This comprehensive report provides an exhaustive, dot-by-dot guide to acquiring the exact datasets, Architectural Decision Records (ADRs), and evaluation frameworks required to assess software architecture and design rationale. The analysis covers the precise extraction of raw ADRs from specific open-source repositories, the programmatic acquisition of pre-compiled academic benchmarks (such as R2ABench, SADU, and Design Rationale datasets), and the implementation of architectural evaluation rubrics (ATAM, SAAM, CBAM, and Script Concordance Testing).Exact Pre-Compiled Architectural DatasetsFor researchers looking to bypass raw repository mining, several pre-compiled datasets and benchmarks have been explicitly designed to evaluate software architecture generation and diagram understanding. The exact locations, structural contents, and acquisition protocols for these datasets represent the foundational layer of modern architectural evaluation.The R2ABench Dataset (Requirement-to-Architecture)R2ABench is a multi-dimensional benchmark designed to evaluate the capability of Large Language Models to generate system architecture diagrams directly from unstructured Product Requirements Documents (PRDs). As software systems evolve, maintaining consistency between changing requirements and architectural artifacts necessitates repeated manual revisions; R2ABench provides a standardized testbed to automate this through generative AI.To acquire this dataset, researchers must navigate directly to the Figshare repository hosting the compiled files. The exact location is accessible via its Digital Object Identifier at https://doi.org/10.6084/m9.figshare.31891372, or through the specific private access link at https://figshare.com/s/3ef18895bd3d6db5b01a. The dataset contains exactly seventeen high-quality data samples extracted from real-world software projects, comprising nine Python projects and eight Java projects across different domains.Each sample in the dataset consists of two core components: the PRD and the expert-curated PlantUML reference code. To parse and utilize this dataset effectively, data engineers must understand that each PRD is strictly structured into six exact sections. The System Introduction outlines the application background, technical framework, and target users to provide overall context. The Core Objectives summarize the primary high-level goals without delving into implementation details. The Functional Features describe the system's core capabilities, strictly using a "when–then" format to define the actions that each entity should perform under particular conditions. The Technical Constraints specify the development frameworks, communication protocols, and runtime environments. Non-functional Requirements dictate response times, reliability, and security parameters. Finally, the System Architecture Description divides the architecture into the Application Layer, Support Layer, and Infrastructure Layer, conveying both the overall distribution and the structured understanding of each layer's role.The SADU Benchmark (Software Architecture Diagram Understanding)The SADU benchmark evaluates vision-language models (VLMs) on their ability to interpret, ground, and reason over software architecture diagrams as structured engineering artifacts rather than generic images. The benchmark includes exactly 2,431 question-answer tasks covering 24 specific QA subtypes. These tasks range from element retrieval (identifying entities, relations, and clusters) to counting and comparison (calculating the number of incoming or outgoing edges) and deep relational reasoning.Each diagram in the SADU dataset is paired with an exact JSON metadata file that provides a machine-readable representation of its content. Step 2: Execute the Python Parsing and JSONL Compilation ScriptOnce the raw files are aggregated in the compiled_adr_dataset directory, they must be parsed to isolate the logical blocks. The user must copy the following Python script and save it as parse_adrs.py. This script utilizes robust regular expressions designed to catch both Markdown header syntax (## Context) and ReStructuredText header syntax (Context\n=======). It outputs a strict JSONL file (adr_dataset.jsonl), representing the exact format required for loading into tools like HuggingFace datasets or feeding into evaluation pipelines.Action Required by User: Save the text below as parse_adrs.py in the same directory where the bash script was executed. Run the script by typing python3 parse_adrs.py in the terminal.Pythonimport os
import re
import json

# Define exact target directories based on the bash script output

DATASET_DIR = "./architecture_dataset_workspace/compiled_adr_dataset"
OUTPUT_FILE = "./architecture_dataset_workspace/adr_dataset.jsonl"

def parse_adr_content(file_path, file_content):
"""
Parses Markdown and RST files to extract core ADR components.
Utilizes regex lookaheads to capture multi-line paragraphs up to the next major header.
""" # Match the header string and capture everything until the next major header or EOF
context_match = re.search(r'(?i)(?:#+\s*Context|Context\n[=\-\~]+)\s*\n(._?)(?=\n(?:#+\s_[A-Z]|[A-Z][a-z]+\n[=\-\~]+)|$)', file_content, re.DOTALL)
    decision_match = re.search(r'(?i)(?:#+\s*Decision|Decision\n[=\-\~]+)\s*\n(.*?)(?=\n(?:#+\s*[A-Z]|[A-Z][a-z]+\n[=\-\~]+)|$)', file_content, re.DOTALL)
consequences_match = re.search(r'(?i)(?:#+\s*Consequences|Consequences\n[=\-\~]+)\s*\n(._?)(?=\n(?:#+\s_[A-Z]|[A-Z][a-z]+\n[=\-\~]+)|$)', file_content, re.DOTALL)
    status_match = re.search(r'(?i)(?:#+\s*Status|Status\n[=\-\~]+)\s*\n(.*?)(?=\n(?:#+\s*[A-Z]|[A-Z][a-z]+\n[=\-\~]+)|$)', file_content, re.DOTALL)

    # Clean the extracted text by stripping leading/trailing whitespace and line breaks
    context = context_match.group(1).strip() if context_match else None
    decision = decision_match.group(1).strip() if decision_match else None
    consequences = consequences_match.group(1).strip() if consequences_match else None
    status = status_match.group(1).strip() if status_match else None

    # Return structured dictionary mapping to exact dot requirements
    return {
        "source_file": os.path.basename(file_path),
        "status": status,
        "context": context,
        "decision": decision,
        "consequences": consequences,
        "raw_text": file_content
    }

def compile_dataset():
parsed_records = []

    # Iterate through the compiled dataset directory
    for filename in os.listdir(DATASET_DIR):
        file_path = os.path.join(DATASET_DIR, filename)
        if os.path.isfile(file_path):
            with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
                content = f.read()
                parsed_data = parse_adr_content(file_path, content)

                # Only append files that contain at least a Decision or Context block to filter out junk
                if parsed_data["decision"] or parsed_data["context"]:
                    parsed_records.append(parsed_data)

    # Write to JSONL format for immediate LLM ingestion
    with open(OUTPUT_FILE, 'w', encoding='utf-8') as out_f:
        for record in parsed_records:
            out_f.write(json.dumps(record) + '\n')

    print(f"Successfully compiled {len(parsed_records)} structured ADRs into {OUTPUT_FILE}")

if **name** == "**main**":
compile_dataset()
Analytical Frameworks for Evaluating Extracted DataAcquiring the data is only the first step; evaluating the performance of LLMs on software architecture design requires strict adherence to standardized evaluation rubrics. The empirical research identifies several specific frameworks that must be programmatically or manually applied to the LLM outputs to determine their viability.The IHUM-Category Rubric for Design RationaleWhen evaluating how well an LLM generates a Design Rationale based on the extracted GitHub issues or Stack Overflow posts, the IHUM-Category classification method must be applied to the generated arguments. This qualitative assessment framework classifies the specific argument points generated by the model into four distinct categories to judge trustworthiness and applicability.Evaluation CategoryDefinition of Output QualityScoring ImplicationInsightful (I)The generated argument reveals deep architectural consequences that humans may have overlooked, demonstrating advanced synthesis.Positive impact on qualitative assessment.Helpful (H)The generated argument accurately reflects known software engineering principles and aligns with standard architectural trade-offs.Considered valid and correct (Baseline expectation).Uncertain (U)The generated argument cannot be definitively proven true or false given the architectural context, or relies on hallucinated constraints.Requires manual human-in-the-loop review.Misleading (M)The generated argument provides factually incorrect assertions about an architectural pattern, which could induce system failures if implemented.Severe penalty in qualitative evaluation.In the evaluation by Zhou et al., the application of this rubric revealed that between 64.45% and 69.42% of LLM-generated arguments not explicitly mentioned by human experts were classified as "Helpful". Conversely, 1.59% to 3.24% of the arguments were classified as "Misleading," indicating that while LLMs can augment architectural documentation, their outputs must be subjected to rigorous constraint checking.Legacy SEI Evaluation Methodologies: QAW, SAAM, ATAM, and CBAMFor an LLM or an automated agent to accurately execute an architectural evaluation on the downloaded ADRs, the prompt engineering pipeline must strictly incorporate the rigorous methodologies developed by the Software Engineering Institute (SEI) at Carnegie Mellon University. These methodologies define the sequential flow of architectural analysis from inception to economic justification.The Quality Attribute Workshop (QAW) is the earliest intervention point, occurring before the software architecture has even been created. It provides a facilitated method that engages system stakeholders to discover the driving quality attributes of a software-intensive system. The QAW process generates "raw scenarios," prioritizes them through a voting process, and refines them into test cases containing explicit stimuli, environmental conditions, and expected responses.Once an initial architecture is drafted, the Software Architecture Analysis Method (SAAM) serves as a lightweight, scenario-based technique to evaluate non-functional quality attributes, with a primary emphasis on modifiability. The process requires stakeholders to develop concise scenarios and classify them as either "direct" (testing existing support without requiring changes) or "indirect" (probing modifications that require architectural changes). For each indirect scenario, the architect must identify the architectural modifications needed, the impacted components, and the estimated cost and effort, allowing for the early detection of design flaws.The Architecture Tradeoff Analysis Method (ATAM) is the comprehensive successor to SAAM, expanding the evaluation to assess how multiple competing quality attributes interact and trade off against one another. The ATAM is a risk-mitigation process that mandates the explicit generation of a "Quality Attribute Utility Tree," which maps core business requirements to specific architectural properties down to the leaf level of scenarios. During the analysis phases, the evaluation team explicitly identifies architectural risks, non-risks, sensitivity points, and tradeoff points. The methodology requires distinct roles—including an Evaluation Leader, Scribe, Timekeeper, and Process Observer—to ensure the rigorous elicitation of constraints.Finally, the Cost Benefit Analysis Method (CBAM) picks up where the ATAM leaves off by adding economic modeling to the technical tradeoffs. Because finite resources govern system creation and maintenance, every architectural choice competes for inclusion. The CBAM elicits cost, schedule, and benefit information from stakeholders to calculate a desirability metric, essentially a ratio of benefit divided by cost. This allows organizations to associate priorities, costs, and benefits with specific architectural decisions, forming a strategic roadmap for software design and evolution.Automated Architectural Decision Frameworks: CAPI and ArchAgentModern architectural evaluation increasingly relies on programmatic decision-support approaches. The Comprehensive Architecture Pattern Integration (CAPI) method utilizes a diagnostic decision tree to suggest architectural patterns depending on user needs. By navigating users through a structured series of questions regarding the envisioned system, CAPI outputs a curated list of architectural design patterns suitable for implementation, effectively replacing manual trial-and-error selection. The method uses a set of 47 distinct patterns evaluated against a decision tree, and recent studies demonstrate that feeding CAPI's decision-tree output into LLM prompts substantially improves the precision of architectural recommendations.For legacy systems that lack comprehensive documentation, the ArchAgent framework provides scalable architecture understanding by integrating static analysis, graph-based reasoning, and LLM-powered semantic synthesis. Because LLMs face severe context limitations—failing to load million-line repositories in a single pass—ArchAgent employs adaptive grouping to partition the codebase into LLM-compatible chunks based on a maximum token threshold. It traces downstream call chains, aggregates cross-repository signals (such as API usage and QPS metrics), and generates functionally meaningful abstractions that accurately represent the system's architecture.Cross-Disciplinary Evaluation: Script Concordance Testing (SCT)Evaluating architectural decision-making requires methodologies capable of measuring reasoning under conditions of extreme uncertainty. The Script Concordance Test (SCT), originally designed to assess clinical reasoning in medical education, provides a highly effective framework for evaluating software design rationale. SCT measures how well an examinee's knowledge is structured by comparing their responses to a panel of experts across a spectrum of ambiguous, evolving scenarios.In an SCT evaluation, the examinee (or an LLM agent) is presented with a scenario and an initial hypothesis; upon receiving new, often confounding information, they must indicate on a five-point Likert scale (ranging from -2 to +2) the extent to which the new data affects the probability of the hypothesis. This methodology acknowledges that expert practitioners often interpret data and respond to uncertain situations in ways that vary within an acceptable range, rather than relying on binary right-or-wrong answers. In the context of the ER-Reason benchmark, SCT-style questions are utilized to assess whether LLMs update their diagnostic beliefs in the correct direction and magnitude as clinical evidence accumulates. Applying this same concordance methodology to architectural decision records allows researchers to rigorously quantify whether an LLM correctly updates its design rationale when confronted with new technical constraints, shifting business requirements, or deprecation notices within the codebase.ConclusionThe empirical acquisition and analysis of software architecture datasets demand strict, grounded methodologies that move beyond abstract theory into executable code. By cloning the exact repositories listed—such as openedx/edx-platform, backstage/backstage, and adr/madr—and deploying the provided Bash and Python extraction scripts, data engineers can reliably assemble a pristine, high-fidelity dataset of real-world Architectural Decision Records. Accessing the exact Figshare links for R2ABench and utilizing the structured JSON metadata schemas of the SADU benchmark provides immediate, programmable access to state-of-the-art multimodal architecture evaluation data. When integrated with formal evaluation rubrics—including the IHUM-Category framework, the SEI's legacy ATAM/SAAM/CBAM procedures, and the uncertainty-focused Script Concordance Test—these artifacts form a complete, reproducible pipeline for advancing empirical software engineering research and generative architecture design.
